1. Introducción
Una vez que las características del problema han sido formuladas y enunciadas de manera clara y concisa, el siguiente paso es implementar el desarrollo del sistema de software siguiendo un modelo de proceso. Todos los modelos de proceso de desarrollo incluyen, de una u otra forma, las siguientes fases: Análisis, Diseño, Implementación, Pruebas y Despliegue. El desarrollo del scanner descrito en este reporte se basa en el estándar IEEE‑830.
1.1 Resumen
Este reporte documenta el ciclo de vida completo de desarrollo de un analizador léxico (scanner) para un lenguaje de programación académico simple con sintaxis similar a C/Java. El scanner lee el código fuente carácter a carácter y clasifica cada secuencia en un token — la unidad mínima con significado (palabra reservada, identificador, literal, operador, etc.). Se presentan dos implementaciones independientes: una construida con la herramienta flex y C99, y otra que utiliza Autómatas Finitos Deterministas (AFD) codificados manualmente en C#/.NET 10. El reporte abarca el análisis de los requisitos léxicos, el diseño de los AFDs y las tablas de transición, los detalles de implementación del archivo lex, y la verificación y validación mediante casos de prueba.
1.2 Notación
A lo largo de este documento se utiliza la siguiente notación y los siguientes modelos.
Máquinas de Estado Finito (AFD). Un Autómata Finito Determinista es una 5-tupla M = (Q, Σ, δ, q0, F) donde:
* Q — conjunto finito de estados
* Σ — alfabeto de entrada (conjunto de caracteres)
* δ: Q × Σ → Q — función de transición
* q0 ∈ Q — estado inicial
* F ⊆ Q — conjunto de estados de aceptación (finales)
En las tablas de transición, el símbolo > marca el estado inicial y * marca los estados de aceptación. ERROR indica una transición inválida (el autómata rechaza la entrada en ese punto). EMIT significa que el autómata ha reconocido un token completo y lo emite, aplicando el principio de máximo consumo (longest match) antes de devolver el control.
Expresiones Regulares. Se utiliza notación estándar: * (estrella de Kleene — cero o más), + (uno o más), ? (cero o uno), | (alternación), [ ] (clase de caracteres), [^ ] (clase negada), \. (punto literal escapado).
Justificación del modelo. Se eligieron los AFDs como modelo de implementación porque garantizan tiempo de escaneo O(n) sin retroceso (backtracking), y son directamente realizables como tablas de transición o cadenas switch-case. Cada categoría de token tiene su propio AFD independiente; un coordinador (el Lexer) selecciona el AFD apropiado según el carácter actual y aplica máximo consumo.
Justificación de la herramienta — flex. La herramienta flex (Fast LEXical analyzer) fue seleccionada para la implementación en C porque genera automáticamente un scanner AFD optimizado a partir de reglas de expresiones regulares escritas en un archivo de especificación .l, reduciendo significativamente el esfuerzo de implementación manual y el riesgo de errores. La implementación en C# codifica los mismos AFDs de forma manual para demostrar la teoría subyacente de manera explícita y proveer validación cruzada del diseño.

2. Análisis
El modelo de análisis es el puente entre la descripción a nivel de sistema y el diseño del sistema. Su enfoque principal está en el qué, no en el cómo: qué entradas y salidas manipula el sistema, qué funciones debe realizar, qué comportamientos exhibe y qué restricciones aplican. El modelo de análisis logra tres objetivos primarios: 1) describir qué requiere el usuario, 2) establecer la base para el diseño, y 3) definir un conjunto de requisitos que puedan ser validados una vez construido el sistema.
2.1 Descripción Léxica Informal
El lenguaje a reconocer es un lenguaje académico simple con sintaxis estilo C/Java. A continuación se describe informalmente cada categoría de token:
* Palabras reservadas. Palabras con significado fijo que no pueden usarse como identificadores: if, else, while, for, int, float, return, void, true, false.
* Identificadores. Una secuencia que comienza con una letra (a–z, A–Z) o guion bajo (_), seguida de cero o más letras, dígitos (0–9) o guiones bajos. El lenguaje distingue mayúsculas de minúsculas (Precio y precio son identificadores distintos).
* Literales enteros. Uno o más dígitos decimales (0–9) sin punto decimal.
* Literales flotantes. Uno o más dígitos decimales, seguidos de un punto literal (.), seguidos de uno o más dígitos decimales. El punto debe estar rodeado de dígitos en ambos lados.
* Operadores aritméticos. Los cuatro operadores básicos: +, -, *, /.
* Operadores relacionales. Operadores de comparación: ==, !=, <, >, <=, >=. Las formas compuestas (==, !=, <=, >=) tienen prioridad sobre las simples mediante máximo consumo.
* Operadores de asignación. Asignación simple = y formas compuestas +=, -=, *=, /=. Las formas compuestas tienen prioridad.
* Símbolos especiales. Delimitadores y puntuación: (, ), {, }, ,, ;, ..
* Comentarios de línea. Comienzan con // y se extienden hasta el final de la línea. La secuencia completa (incluyendo el prefijo //) es un único token y no se interpreta más.
* Espacios en blanco. Espacios, tabulaciones y retornos de carro se consumen silenciosamente; separan tokens pero no generan ninguno.
* Errores léxicos. Cualquier carácter no reconocido por las reglas anteriores es un símbolo inválido. Se reporta como error y se descarta (recuperación en modo pánico); el escaneo continúa con el siguiente carácter.

	3.7 Diseño de la Tabla de Símbolos
La tabla de símbolos registra cada identificador y literal numérico encontrado durante el escaneo. Su propósito es proveer al parser y a las fases posteriores del compilador con un registro único y sin duplicados de los símbolos del programa. No almacena palabras reservadas, operadores ni puntuación.
Cada entrada en la tabla contiene cuatro campos: un índice secuencial, la cadena del lexema (texto exacto del código fuente), el tipo de token (ID, INTEGER o FLOAT) y el número de línea de la primera aparición. La implementación usa un arreglo de tamaño fijo de hasta 512 entradas con deduplicación por búsqueda lineal: antes de insertar, se recorre el arreglo para verificar si el lexema ya existe; si es así, no se crea una nueva entrada.
Pseudocódigo de la función de inserción:
 
función sym_insert(lexema, tipo, linea):
        para i en 0 .. contador - 1:
            si tabla[i].lexema == lexema:
                retornar              // ya existe — no duplicar
        tabla[contador].lexema = lexema
        tabla[contador].tipo   = tipo
        tabla[contador].linea  = linea
        contador++

4. Implementación
La implementación se deriva directamente del diseño. La implementación en C utiliza la herramienta flex, que genera un scanner en C a partir del archivo de especificación scanner.l. Un archivo .l se divide en tres secciones separadas por marcadores %%: la Sección de Definiciones, la Sección de Reglas y la Sección de Código de Usuario.
4.1 Sección de Definiciones (scanner.l, líneas 1–78)
La sección de definiciones prepara todo lo que las reglas necesitarán:
* Prólogo C (bloque %{ %}). Contiene directivas #include para stdio.h, string.h y symbol_table.h (que incluye transitivamente token_ids.h). También declara los prototipos de emit() y lex_error().
* Opciones de flex.
   * %option noyywrap — indica a flex que no se necesita la función yywrap() al llegar al EOF. Requerido al compilar sin -lfl.
   * %option yylineno — instruye a flex para que mantenga automáticamente la variable global yylineno, incrementándola en cada nueva línea.
* Macros con nombre. Los fragmentos de expresiones regulares se definen como macros con nombre para mejorar la legibilidad. Se referencian en la Sección de Reglas con la sintaxis {NOMBRE}:
·            
·           LETTER   [a-zA-Z_]
·           DIGIT        [0-9]
·           ID           {LETTER}({LETTER}|{DIGIT})*
·           INT          {DIGIT}+
·           FLOAT        {DIGIT}+\.{DIGIT}+
·           COMMENT  \/\/[^\n]*
BLANK        [ \t\r]+
Nota importante: flex no permite comentarios /* */ en línea en las líneas de %option ni en las definiciones de macros — los trata como parte de la cadena de opción/definición y causa errores de análisis. El archivo original contenía dichos comentarios y requirió corrección antes de que la compilación pudiera realizarse con éxito.
4.2 Sección de Reglas (scanner.l, líneas 80–172)
La sección de reglas (entre los dos marcadores %%) contiene pares patrón–acción. Flex lee cada patrón como una expresión regular; cuando se encuentra una coincidencia, se ejecuta la acción C asociada. El orden de las reglas es crítico, ya que flex resuelve empates en longitud de coincidencia seleccionando la primera regla que coincide:
1. Palabras reservadas primero — evita que if, while, etc. sean clasificados como IDENTIFIER.
2. FLOAT antes que INT — máximo consumo: 3.14 debe consumirse como un único token, no dividirse en 3, ., 14.
3. Operadores compuestos antes que simples — += debe reconocerse antes de que + y = sean reconocidos por separado.
4. Nueva línea (\n) consumida silenciosamente — yylineno ya fue incrementado por flex; no se emite ningún token.
5. Espacios en blanco (BLANK) consumidos silenciosamente — espacios y tabulaciones son separadores de tokens, no tokens en sí.
6. Punto (.) como última regla — el punto de flex coincide con cualquier carácter excepto nueva línea; colocado al final, actúa como captura general para errores léxicos.
Cada regla de palabra reservada y operador llama a emit(token_id, nombre, yytext). Las reglas de identificadores y literales además llaman a sym_insert(yytext, tipo, yylineno) para registrar el símbolo.
4.3 Sección de Código de Usuario (scanner.l, líneas 175–199)
La sección de código de usuario (después del segundo %%) define las dos funciones auxiliares utilizadas en las reglas:
* emit(int id, const char* nombre, const char* lexema) — imprime una fila de la tabla de tokens en stdout usando printf con especificadores de formato de ancho fijo para el Token ID, el nombre descriptivo, el texto del lexema y el número de línea actual (yylineno).
* lex_error(const char* mal_caracter) — imprime un mensaje de error en stderr indicando el número de línea y el carácter no reconocido. La estrategia de recuperación es modo pánico: el carácter problemático se descarta silenciosamente y el scanner continúa con el siguiente carácter de entrada, asegurando que un único símbolo incorrecto no aborte el escaneo completo.

5.4 Validación Cruzada de AFDs en C#
La implementación en C# valida cada token emitido contra su expresión regular .NET equivalente usando Regex.IsMatch() inmediatamente después de que el AFD lo acepta. Para la entrada de referencia anterior, los 9 tokens distintos de espacios en blanco retornaron True. Esto confirma que los AFDs codificados manualmente son consistentes con las especificaciones de expresiones regulares formales definidas en la sección de Análisis.
